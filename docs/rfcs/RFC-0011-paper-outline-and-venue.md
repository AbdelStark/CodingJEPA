# RFC-0011 — Paper outline and target venue

## Status
Locked (2026-05-15)

## Problem

Specify what we write up, where we send it, and what would have to be true in `results/RESULTS-MEMO.md` for submission to be defensible.

## Decisions locked

### D1 — Target venue (in priority order)
1. **NeurIPS workshop** on representation learning or code intelligence (e.g., NeurIPS R0-FoMo, NeurIPS ML4Code if it returns).
2. **ICLR workshop** on representation learning.
3. **MSR / ASE / FSE** tool track or short-paper track.
4. **arXiv-only** with a polished blog post if the metric bar is met but the venue fit is poor.

We target a **workshop or short paper**, not a main-track full paper, because the contribution is empirical and the scope is deliberately narrow. We do not pretend a single-GPU artifact is a flagship result.

### D2 — Paper length
- 8 pages + references for workshop format; 4 pages for short-paper venues.
- Numbers, tables, and figures are owned by `results/results.json` and `results/figures/`. The paper draft references those artifacts.

### D3 — Title (working)
**"CodingJEPA: A Joint-Embedding Predictive Architecture for Python Code Refactoring."**

### D4 — Outline

1. **Abstract.** Thesis + headline numbers + matched-compute caveat. ≤ 200 words.
2. **Introduction.**
   - Why code refactoring is a poor fit for token-level objectives.
   - Why JEPA + a distributional regularizer is plausible.
   - One-sentence inheritance from LeWorldModel.
   - Bulleted contributions: (a) recipe transfer, (b) intent conditioning, (c) results, (d) released artifacts.
3. **Related work.**
   - JEPA family (I-JEPA, V-JEPA, LeWorldModel).
   - Code representation learning (CodeBERT, GraphCodeBERT, UniXcoder, CodeT5, CodeRetriever).
   - Refactor mining (RefactoringMiner, PyDriller).
4. **Method.**
   - 4.1 Architecture (RFC-0003 in 1 page).
   - 4.2 Training objective (RFC-0008 §D3 in ½ page).
   - 4.3 Intent conditioning (RFC-0004 in ½ page).
   - 4.4 Retrieval-rerank inference (RFC-0009 in ½ page).
5. **Data.**
   - 5.1 Source repos (cite `docs/data/CANDIDATE_REPOS.md`).
   - 5.2 Pair mining and intent labeling (RFC-0002 condensed).
   - 5.3 Splits and dedup (RFC-0014 condensed).
6. **Experiments.**
   - 6.1 Baselines (RFC-0005 §D5).
   - 6.2 Main results (CJ-RET-100, CJ-RET-1k, CJ-EXEC) with paired bootstrap p-values.
   - 6.3 Per-intent table.
   - 6.4 Robustness probes.
   - 6.5 OOD probe.
   - 6.6 Ablations: SIGReg, intent, projector, history `H`, λ sweep.
7. **Analysis.**
   - Embedding-space visualizations (t-SNE / UMAP coloured by intent and by repo).
   - Failure-mode qualitative examples (1 page).
   - When CodingJEPA loses to CodeBERT and why.
8. **Limitations.**
   - Single language (Python).
   - Small model.
   - Heuristic intent labels.
   - Public-corpus contamination caveat.
   - No generative head.
9. **Conclusion.** 1 paragraph.
10. **Reproducibility statement.** Pointer to checkpoint, manifest, `make eval`.
11. **Appendix.**
    - A — Full hyperparameter tables.
    - B — Per-intent acceptance rules (verbatim from RFC-0004).
    - C — Robustness perturber details.
    - D — Confusion gallery (8 pages).
    - E — Additional ablations (history `H = 1, 3, 6`; encoder size).

### D5 — Submission preconditions
The paper is submission-ready only when:
- PRD §8 numeric bars are met on the test split;
- the gold subset shows κ ≥ 0.7 inter-rater on human review;
- every figure and table in the paper is generated by `make eval` from the released artifacts;
- the limitations section is at least 1 page of honest, technical caveats — not boilerplate.

### D6 — Authorship and ethics
- Author list and order finalized before submission.
- Ethics statement covers: source-code licensing (BSD-3 attribution), data redistribution, dual-use risks (auto-refactor tools can introduce subtle bugs; we list the safety rails from RFC-0007).
- Conflicts of interest: declared in the paper.

### D7 — Reproducibility package (CR companion)
- Checkpoints on Hugging Face Hub (`hf://<org>/coding-jepa-v1`).
- Data manifest + recipes to re-mine pairs.
- `make eval` command and a Docker image pinned by digest.
- The eval container is reproducible from `Dockerfile.eval` and the lockfiles in the repo.

### D8 — Negative-result protocol
If PRD §8 bars are *not* met:
- We write a short technical report (4 pages) titled **"What Doesn't Work: JEPA on Python Refactoring at Single-GPU Scale"** with the same artifacts.
- Honest failure is more valuable to the community than silence.

## Deferred items
- Conference full-paper submission.
- Blog post / project page polish.
- A tutorial-format workshop paper for ML4Code.

## Acceptance condition

Locked when:
- the paper draft `paper/main.tex` exists with the section structure above (even if empty);
- the figure-generation scripts emit PDFs to `results/figures/` from `make eval`;
- the venue list is acknowledged by the project lead.

# v1.0.0 release runbook

Operator-readable script implementing the 8-step release procedure in
`docs/spec/09-release-and-versioning.md` §Release procedure. Run from a
fresh clone of `main`. The same procedure applies to `v1.x.y` releases;
substitute the version number throughout.

> **Dry-run on `rc.0`.** Step through every command below against an
> `rc.0` candidate before signing the actual release tag. The acceptance
> bullet for this issue is "a dry-run of every step succeeds on `rc.0`".

## Pre-conditions

- You are a maintainer with push rights to `main`.
- `gh auth status` is green.
- `gpg --list-secret-keys` shows the maintainer signing key (RFC-0013 §D8).
- `HF_TOKEN` is exported and scoped to the release HF repos.
- `docker login ghcr.io` succeeded.
- `uv` is installed; `python --version` is 3.12.

## Step 1 — Pre-flight checks

All CI gates green on `main`:

- `lint` workflow (#18)
- `unit` workflow (#19)
- `safety` workflow (#20)
- `eval-smoke` workflow (#21)
- `perf` workflow (#22)
- `changelog` workflow (#27)
- Nightly diagnostics (#23): rank, leakage, intent balance — green for **≥ 3 consecutive nights**.

```bash
gh run list --workflow=lint --branch=main --limit=1 --json conclusion -q '.[0].conclusion'      # success
gh run list --workflow=unit --branch=main --limit=1 --json conclusion -q '.[0].conclusion'      # success
gh run list --workflow=safety --branch=main --limit=1 --json conclusion -q '.[0].conclusion'    # success
gh run list --workflow=eval-smoke --branch=main --limit=1 --json conclusion -q '.[0].conclusion'  # success
gh run list --workflow=perf --branch=main --limit=1 --json conclusion -q '.[0].conclusion'      # success
gh run list --workflow=nightly --branch=main --limit=3 --json conclusion -q '.[].conclusion'    # success x3
```

If any check fails: stop, fix on a PR, re-run.

## Step 2 — Eval reproduction

```bash
git checkout main && git pull
uv sync --frozen
make eval                          # produces results/results.json
diff <(jq -S . results/results.json) <(jq -S . results/results.lock.json)
```

The numbers must match the committed `results/results.lock.json` within
bootstrap CI overlap (per spec/09 §Reproducibility guarantees). If `diff`
shows changes that exceed CI overlap, stop — the release reproducibility
contract is broken.

## Step 3 — Versions and changelogs

```bash
# Bump pyproject.toml — single edit.
sed -i 's/^version = "0\\.0\\.0"$/version = "1.0.0"/' pyproject.toml

# Move [Unreleased] content under the new version heading.
python tools/changelog_release.py 1.0.0  # tool to be added when this step is exercised; for now,
                                          # do the equivalent edit manually:
#   - copy bullets under [Unreleased] to a new [1.0.0] - YYYY-MM-DD heading
#   - leave an empty [Unreleased] block above it
#   - update the comparison link at the bottom

# Verify lint still passes.
make lint
```

Commit on a PR titled `release: v1.0.0`. Wait for CI to go green. Merge.

## Step 4 — Tag

Signed tag, never from a fork:

```bash
git checkout main && git pull
git tag v1.0.0 -s -m "v1.0.0"
git push origin v1.0.0
```

If the tag command fails with GPG errors: fix the signing key, do not bypass with `--no-gpg-sign`.

## Step 5 — Build

```bash
uv build                                        # wheels in dist/
docker build -t coding-jepa:v1.0.0-eval -f Dockerfile.eval .
```

Both must be reproducible from the same tag commit. If `git status` shows any
untracked files inside the build context, stop and clean.

## Step 6 — Publish

### Wheels → PyPI

```bash
uv publish --token "$PYPI_TOKEN" dist/*
```

### Eval image → GHCR

```bash
docker tag coding-jepa:v1.0.0-eval ghcr.io/abdelstark/coding-jepa:v1.0.0-eval
docker push ghcr.io/abdelstark/coding-jepa:v1.0.0-eval

# Capture the digest the next step needs.
docker inspect ghcr.io/abdelstark/coding-jepa:v1.0.0-eval --format='{{index .RepoDigests 0}}'
```

Record the digest in `MODEL_CARD.md` (the `tools/model_card_update.py`
script from #125 handles this when given a digest manifest).

### Model weights + tokenizer + MODEL_CARD.md → HF Hub

```bash
python tools/model_card_update.py path/to/checkpoint.safetensors data/manifest.lock.json \\
    --tokenizer tokenizer/v1/tokenizer.model \\
    --training-compute-h100-hours <N> \\
    --seeds-reported 3

huggingface-cli upload abdelstark/coding-jepa-v1 \\
    encoder.safetensors projector.safetensors predictor.safetensors pred_proj.safetensors \\
    tokenizer/v1/tokenizer.model tokenizer/v1/special_tokens.json \\
    MODEL_CARD.md
```

### Pairs corpus → HF Datasets

```bash
huggingface-cli upload abdelstark/coding-jepa-v1-pairs \\
    data/pairs/v1.parquet data/splits/v1.lock.json data/audit/dedup.json
```

License-compliance check (RFC-0014 §D2): the pairs corpus must carry the
most-restrictive constituent license (BSD-3-Clause). The HF Dataset
README captures this.

## Step 7 — Release notes

```bash
gh release create v1.0.0 \\
    --title "v1.0.0" \\
    --notes-file release-notes/v1.0.0.md \\
    dist/*.whl
```

`release-notes/v1.0.0.md` links to:

- The `[1.0.0]` section of `CHANGELOG.md` (deep link).
- The HF Hub release at `abdelstark/coding-jepa-v1`.
- The GHCR digest from step 6.
- The relevant tracking issues (#3 infra, #4 data, #5 model, #6 training, #7 baselines, #8 inference, #9 safety, #10 demo, #11 eval, #12 release).

## Step 8 — Audit trail

`MODEL_CARD.md` is the durable identifier. After step 6 it carries:

- `checkpoint_hash` — sha256 of the released weight blob.
- `manifest_hash` — sha256 of the canonicalized `data/manifest.lock.json`.
- `tokenizer_hash` — sha256 of `tokenizer/v1/tokenizer.model`.
- `index_id` — `checkpoint_hash[:8]-manifest_hash[:8]`.
- `training_compute_h100_hours`.
- `seeds_reported` — must be ≥ 3 for the release to be non-preliminary
  (RFC-0005 §D7).

The reproducibility statement in the paper points at `git tag v1.0.0`.

## Yank procedure

If a release inadvertently breaks reproducibility (spec/09 §Reproducibility
guarantees):

```bash
git tag -d v1.0.0 && git push --delete origin v1.0.0
gh release delete v1.0.0 --yes
huggingface-cli repo delete abdelstark/coding-jepa-v1 --type model --yes  # only if necessary
docker manifest rm ghcr.io/abdelstark/coding-jepa:v1.0.0-eval || true
```

Open a yank-and-re-issue tracking issue and re-cut.

## Post-release

- Open a `release/v1.0.x` branch from the tag (for future patch backports).
- Open the **next** `[Unreleased]` block in `CHANGELOG.md` on `main`.
- Update the support-window block in `docs/spec/09-release-and-versioning.md`
  with the v1.0.0 release date for the 6-month patch window.
- Close any release-tracking issues (#12 dashboard).
